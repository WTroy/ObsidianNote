学习⼤数据技术时需要关注的重点：

HDFS 理论知识点：
存储模型
⽂件线性按字节切割成块 (block) ，具有 offset ， id
⽂件与⽂件的 block ⼤⼩可以不⼀样
⼀个⽂件的 block ，除最后⼀个 block ，其他 block ⼤⼩⼀致
block 的⼤⼩根据硬件的 I/O 特性调整
block 被分散存放在集群的节点中，具有 location
Block 具有副本 replication ，没有主从概念，副本不能出现在同⼀个节点
副本是满⾜可靠性和性能的关键
⽂件上传可以指定 block ⼤⼩和副本数，上传后只能修改副本数
⼀次写⼊多次读取，不⽀持修改
⽀持追加数据
架构设计
HDFS 是⼀个主从 (master/slave) 架构
由⼀个 NameNode 和若⼲个 DataNode 组成
⾯向⽂件包含：⽂件数据 (data) 和⽂件元数据 (metadata)
NameNode 负责存储和管理⽂件元数据，并维护了⼀个层次型的⽂件⽬录树
DataNode 负责存储⽂件数据 (block) ，并提供 block 的读写
DataNode 和 NameNode 维持⼼跳，并汇报⾃⼰持有的 block 信息
Client 和 NameNode 交互⽂件元数据和 DataNode 交互⽂件 blick 数据

⻆⾊功能
NameNode
完全基于内存存储⽂件元数据，⽬录结构，⽂件 block 的映射
需要持久化⽅案保证数据可靠性
提供副本放置策略
DataNode
基于本地磁盘存储 block( ⽂件的形式 )
并保存 block 的校验和数据保证 block 的可靠性
与 NameNode 保持⼼跳，汇报 block 列表状态
元数据持久化
任何对⽂件系统元数据产⽣修改的操作， NameNode 都会使⽤ editlog 的事务⽇志记录下来
使⽤ FsImage 存储内存所有的元数据状态
使⽤本地磁盘保存 editlog 和 FsImage
editlog 具有完整性，数据丢失少，但恢复速度慢，并有体积膨胀⻛险
FsImage 具有恢复速度快，体积和内存数据相当，但不能实时保存，数据丢失多
NameNode 使⽤ editlog+FsImage 整合的⽅案：
滚动将增量的 editlog 更新到 FsImage ，以保证更近点的 FsImage 和更⼩的 editlog 体积
HDFS 搭建时会格式化，格式化操作会产⽣⼀个空的 FsImage
当 NameNode 启动时，它从硬盘中读取 editlog 和 FsImage
将所有的 editlog 中的事务作⽤到内存中的 FsImage 上

持久化扩展：
并将这个新版本的 FsImage 从内存中保存到本地磁盘上
然后删除旧的 editlog ，因为这个旧的 editlog 的事务都已经作⽤在 FsImage 上了
HDFS 中的 SNN(SecondaryNameNode)-- ⾼可⽤模式下不⽤
在⾮ HA 模式下， SNN ⼀般是独⽴的节点，周期完成对 NameNode 的 editlog 和 FsImage 合并，减少 editlog ⼤⼩，减少
NN 的启动时间
根据配置⽂件设置的时间间隔 fs.checkpoint.period, 默认 3600 秒
根据配置⽂件设置 editlog 的⼤⼩  fs.checkpoint.size 规定， editlog ⽂件的最⼤值默认是 64MB
安全模式
NameNode 启动后会进⼊⼀个称为安全模式的特殊状态
处于安全模式的 NameNode 不会进⾏数据块的复制
NameNode 从所有的 DataNode 接收⼼跳信号和块状态报告
每当 NameNode 发现某个数据块的副本数⽬达到这个最⼩值，那么认为该数据块是副本安全的 (safely replication)
在⼀定百分⽐ ( 这个参数可配置 ) 的数据块被 NameNode 检测确认是安全之后 ( 加上⼀个额外的 30 秒等待时
间 ),NameNode 将退出安全模式状态
接下来它会确认还有哪些数据块的副本数量没有达到指定数⽬，并将这些数据块复制到其他的 DataNode 上
Block 副本放置策略
第⼀个副本：放置在上传⽂件的 DataNode ：如果是集群外提交，则随机挑选⼀台磁盘不太满， CPU 不太忙的节点

第⼆个副本：放置在与第⼀个副本不同机架的节点上
第三个副本：与第⼆个副本相同机架的节点
更多副本：随机节点
读写流程
HDFS 写流程
Client 与 NN 建⽴连接并创建⽂件元数据
NN 判定元数据是否有效
NN 触发副本放置策略，返回⼀个有序的 DN 列表
Client 和 DN 建⽴ PipeLine 连接
Client 将块切分成 package(64kb) ，并使⽤ chunk(512B)+chunksum(4B) 填充
Client 将 package 放⼊发送队列 dataqueue 中，并向第⼀个 DN 发送
第⼀个 DN 接收到 package 后保存本地并发送给第⼆个 DN
第⼆个 DN 接收到 package 后保存本地并发送给第三个 DN
在这个过程中，上游节点同时发送下⼀个 package
类⽐⼯⼚的流⽔线。结论：流式其实也是变种的并⾏计算
HDFS 使⽤这种传输⽅式，副本数对于 client 是透明的
当 block 传输完成， DN 们各⾃向 NN 汇报，同时 client 继续传输下⼀个 block
所以， client 的传输和 block 的汇报也是并⾏的
HDFS 读流程
为了降低整体的带宽消耗和读取延时， HDFS 会尽量让读取程序读取离他最近的副本
如果在读取程序的同⼀个机架上有⼀个副本，那么就读取该副本
如果⼀个 HDFS 集群跨越多个数据中⼼，那么客户也将⾸先读本地数据中⼼的副本
语义：下载⼀个⽂件
client 和 NN 交互⽂件元数据获取 fileBlockLocation
NN 会按照距离策略排序返回

client 尝试下载 block 并校验数据完整性
语义：下载⼀个⽂件其实是获取⽂件的所有 block 元数据，那么⼦集获取某些 block 应该成⽴
HDFS ⽀持 client 给出⽂件的 offset ⾃定义连接哪些 block 的 DN ，⾃定义获取数据
这个是⽀持计算层的分治，并⾏计算的核⼼
