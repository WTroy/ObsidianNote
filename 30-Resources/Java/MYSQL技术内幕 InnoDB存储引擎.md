InnoDB 存储引擎⽀持事务，其设计⽬标主要是⾯向在线事务处理 (OLTP) 的应⽤。从 Mysql 5.5.8 版本开始， InnoDB 是默

对于表中的数据， InnoDB 采⽤了聚集 (clustered) 的⽅式，因此每张表的存储都是按主键的顺序进⾏存放。如果没有显

数据库中的缓冲池是通过 LRU(Latest Recent Used) 算法来进⾏管理的。
在 InnoDB 存储引擎中，缓冲池中⻚的⼤⼩默认是 16KB ，同样使⽤ LRU 算法进⾏管理。但是稍微做了⼀些算法的优化：
在 LRU 列表中加⼊了 midpoint 位置，新读取的⻚虽然是最新访问的⻚，但是并不直接放到 LRU 列表的⾸部，⽽是放到
LRU 列表的 midpoint 位置。
如果直接将新读取的数据直接放到 LRU 的⾸部，某些 sql 会导致缓冲池中的⻚被刷新出，从⽽影响缓冲池的效率。⽐如：
索引或数据的扫描操作，会读取很多⻚甚⾄全部⻚。为了解决这个问题，引⼊了⼀个参数来进⼀步管理 LRU 列表，这个
参数是innodb_old_blocks_time，⽤于表示读取到 mid 位置后需要等待多久才会被加⼊到 LRU 列表的热端。
但是当数据库刚启动的时候， LRU 列表为空，没有任何的⻚。这时候⻚都存放在Free 列表中。当需要从缓冲池中分区
时，⾸先从 Free 列表中查找是否有可⽤的空闲⻚，若有则将该⻚从 Free 列表中删除，放⼊到 LRU 列表中。否则，根据
LRU 算法，淘汰 LRU 列表末尾的⻚，将该内存空间分配给新的⻚。
