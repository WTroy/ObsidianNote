zookeeper 是什么？
zookeeper 是源代码开放的分布式协调服务，由雅⻁创建，是 Google Chubby 的开源实现。
zookeeper 是⼀个⾼性能的分布式数据⼀致性解决⽅案，它将那些复杂的，容易出错的分布式⼀致性服务封装起来，构
成⼀个⾼效可靠的源语集，并提供⼀系列简单易⽤的接⼝给⽤户使⽤。

数据发布 / 订阅
数据发布 / 订阅，顾名思义就是⼀⽅把数据发布出来，另⼀⽅通过某种⼿段可以得到这些数据
通常数据订阅有两种⽅式：推模式和拉模式。推模式⼀般是服务器主动向客户端推送消息；拉模式是客户端主
动去服务器获取数据（通常采⽤的是定时轮训的⽅式）
zk 采⽤的是两者相结合：发布者将数据发不到 zk 集群节点上，订阅者通过⼀定的⽅法告诉服务器，我对哪个节
点感兴趣，那服务器在这些节点的数据发⽣变化时，就通知客户端，客户端得到通知后可以去服务器获取数据
信息。

时节点。临时节点在服务器出现问题的时候，节点会⾃动的从 zk 上删除，那么 zk 上的服务器列表就是最新的可

会话：
会话是指客户端和 zookeeper 服务器的连接， zookeeper 中的会话叫 session ，客户端与服务器建⽴⼀个 TCP 的⻓连接来
维持⼀个 session ，客户端在启动的时候⾸先会与服务器建⽴⼀个 TCP 连接，通过这个连接，客户端能够通过⼼跳检测
与服务器保持有效的会话，也能向 zk 服务器发送请求并获得响应。
数据节点：
zookeeper 中的节点有两类：
命名服务：顾名思义，就是提供名称的服务，例如数据库表格 ID ，⼀般⽤的⽐较多的有两种：⼀种是⾃增⻓ ID ，⼀
种是 UUID 。两种 ID 都各有缺陷，⾃增⻓只能在但表中使⽤，不能在分布式中使⽤， UUID 可以在分布式中使⽤但是
ID 没有规律难以理解。
我们可以借助 zk 来⽣成⼀个顺序增⻓的，可以在集群环境中使⽤的，命名易于理解的 ID 。
分布式协调 / 通知
⼼跳检测：在分布式系统中，我们常常需要知道某个机器是否可⽤，传统的开发中，通过 ping 某个机器来实
现， ping 通说明对⽅是可⽤的，相反则不可⽤。 zk 中我们让所有的机器都注册了⼀个临时节点，我们判断⼀个
机器是否可⽤，只需要判断这个节点在 zk 中是否存在即可，不需要直接去连接需要检查的机器，降低系统的复
杂度。

悲观锁：⼜叫悲观并发锁，是数据库中⼀种⾮常严格的策略，具有强烈的排他性，能够避免不同事务对同⼀数据并
发更新造成的数据不⼀致性，在上⼀个事务没有完成之前，下⼀个事务不能访问相同的资源，适合数据更新竞争⾮
常激烈的场景
乐观锁：相对于悲观锁，使⽤场景很更多。乐观锁认可竞争的存在，但是在⼤多数场景下不同事务访问相同资源很
少出现相互⼲扰的情况，因此在事务处理期间不需要进⾏并发控制。乐观锁也是锁，它还是会有并发控制。
对于数据库我们通常做法是在每个表中增加⼀个 version 字段，事务修改数据钱先读出数据，必然顺势读出
version 。然后把这个 version 作为更新的必要条件，那么更新失败说明其他事务已经修改过数据了，那系统需要
抛出异常给客户端，让客户端⾃⾏处理，客户端可以选择重试。

ACL 权限控制：

CREATE ：创建⼦节点的权限
READ ：获取节点数据和⼦节点列表的权限
WRITE ：更新节点数据的权限
DELETE ：删除⼦节点的权限
ADMIN ：设置节点 ACL 的权限
